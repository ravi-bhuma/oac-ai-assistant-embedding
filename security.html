Module 02

Security Deep-Dive

A detailed walkthrough of both authentication models for embedded OAC content — browser-based (session) and token-based (OAuth 2.0) — including IDCS configuration, token lifecycle, and production deployment patterns.

Security is the most critical aspect of embedding analytics. Every embedded query that reaches OAC must be authenticated and authorized. Oracle Analytics supports two distinct authentication models, each with different trade-offs for user experience, complexity, and browser compatibility.

This module walks through both models in detail, explains the underlying OAuth 2.0 mechanics, and gives you the exact steps to configure each one.

Authentication Model A

Browser-Based (Session) Authentication

How It Works

Browser-based authentication is the simplest embedding model. It relies on the user already having an active OAC session in their browser. When the embedded page loads, the embedding framework piggybacks on the existing session cookie to authenticate API calls back to OAC.

User ➔ Logs into OAC
(or SSO handles it) ➔ Session Cookie
set in browser
User ➔ Opens embedded page ➔ embedding.js loads ➔ Cookie sent
with API calls ➔ OAC validates
session

Step-by-Step Setup

Step A-1

Ensure Users Have OAC Accounts & Permissions

Every user who will access the embedded page must have an OAC user account with the Use Assistant in Workbooks permission. This permission is included by default in the DV Content Author application role. If you need tighter control, create a custom application role and explicitly grant the permission.

Verify this in the OAC Console under Users and Roles.

Step A-2

Configure SSO (If Not Already Done)

For a seamless experience, your organization should have SSO configured between your identity provider (Azure AD, Okta, ADFS, etc.) and Oracle Identity Cloud Service (IDCS) / OCI IAM Identity Domains. When SSO is in place, users authenticate once and gain access to both your portal and OAC without a second login prompt.

Without SSO, users must manually log into OAC in a separate browser tab before the embedded page will render. The session cookie is shared across tabs within the same browser profile.

Step A-3

Register the Embedding Origin as a Safe Domain

OAC enforces CORS restrictions on embedded content. You must explicitly register the hosting domain in OAC's Safe Domains configuration:

  1. Sign in to OAC as an administrator.
  2. Navigate to Console → Administration → Safe Domains.
  3. Click Add Domain.
  4. Enter the exact origin of your embedding page (e.g., https://portal.yourcompany.com or http://localhost:8000 for local testing).
  5. Check the Embedding checkbox to allow OAC components to render on pages from this origin.
  6. Save.
❌
The origin must match exactly — protocol (http vs https), hostname, and port. http://localhost:8000 and http://localhost:5500 are different origins. A mismatch will produce CORS errors in the browser console.
Step A-4

Use the Standard Embedding Code (No Token Config)

With browser-based auth, no special security configuration is needed in your JavaScript. The standard applyBindings() call is sufficient:

<oracle-analytics-assistant
  id="oaAssistant"
  src-data="[&quot;XSA('user'.'dataset')&quot;]">
</oracle-analytics-assistant>
<script>
  oracle.oa.embedding.ready().then(oApp => oApp.applyBindings());
</script>

That's it. The browser's session cookie handles authentication automatically.

When to Use Browser-Based Auth

  • Internal portals and intranets where all users have OAC accounts.
  • Environments with SSO already configured (Azure AD, Okta, ADFS + IDCS).
  • Rapid prototyping and development — it's the fastest way to test embedding.
  • Scenarios where you want per-user identity — each user sees data according to their own OAC role and row-level security rules.

Limitations

  • Third-party cookie blocking: Modern browsers (Safari, Chrome with partitioned cookies) increasingly block third-party cookies. If the embedding page and OAC are on different domains, the session cookie may be blocked. A vanity URL (matching domains) or token-based auth resolves this.
  • Login dependency: Without SSO, users must separately log into OAC — adding friction.
  • Not suitable for public/anonymous access: Every viewer needs an OAC account.
Authentication Model B

Token-Based Authentication (OAuth 2.0)

How It Works

Token-based authentication uses an OAuth 2.0 bearer token to authenticate the embedded content. Instead of relying on the user's browser session, a server-side process obtains an access token from Oracle Identity Cloud Service (IDCS) and injects it into the page. The embedding framework uses this token for all API calls to OAC.

This is the recommended model for production deployments, especially for public-facing sites, portals where users don't have OAC accounts, or any scenario where third-party cookie blocking is a concern.

User ➔ Visits embedded page ➔ Page calls
server-side API
Server-side API ➔ Calls IDCS
/oauth2/v1/token ➔ Access Token
returned
Token injected
into page JS ➔ setEmbeddingConfig()
with token ➔ OAC validates
token ➔ AI Assistant
renders
⚠️
Critical security principle: Never expose the IDCS Client ID, Client Secret, or user credentials in client-side JavaScript. Token generation must happen server-side (via Oracle Functions, API Gateway, a backend API, or at minimum a secure server script). The client page only ever receives the opaque bearer token.

Step-by-Step Setup

Step B-1

Create an IDCS Confidential Application

A confidential application in IDCS acts as a proxy between your embedding page and OAC. It holds the Client ID and Client Secret used to request tokens.

  1. Sign in to the OCI Console as an administrator.
  2. Navigate to Identity & Security → Domains (or Federation → OracleIdentityCloudService for older tenancies).
  3. Open the IDCS admin console.
  4. Go to Applications → Add → Confidential Application.
  5. Enter a name (e.g., OAC_Embedding_Token_App) and click Next.
  6. Select Configure this application as a client now.
  7. Under Allowed Grant Types, select:
    • Resource Owner Password Credentials — for server-to-server token requests using a service account.
    • Optionally, Authorization Code — if you plan to use user-interactive OAuth flows (e.g., within APEX).
    • Optionally, Refresh Token — to enable automatic token renewal.
  8. Click Next.
Step B-2

Map the OAC Instance as a Resource (Add Scope)

Scroll down in the application configuration to the Token Issuance Policy section and click Add Scope. This maps your OAC instance as a resource the token can access.

  1. In the "Add Scope" dialog, you'll see a list of IDCS applications. Select the application that corresponds to your OAC instance.
  2. This registers the OAC scope (e.g., https://<oac-instance>...urn:opc:resource:consumer::all) in the confidential app.
  3. Copy the scope value — you will need it when making token requests.
  4. Click Save and Activate the application.

After activation, note the Client ID and Client Secret from the application's Configuration tab. Store these securely — they are equivalent to a password.

Step B-3

Create a Minimally-Privileged Service Account

For the Resource Owner Password Credentials grant type, you need a user account whose credentials will be used to obtain tokens. Best practice:

  • Create a dedicated OAC user (e.g., embedding-svc@yourcompany.com).
  • Assign only the minimum required OAC role — typically DV Content Author or a custom role with only the Use Assistant in Workbooks permission.
  • Grant access only to the specific datasets that should be publicly visible.
  • Never use a personal admin account for token generation.
Step B-4

Generate an Access Token (Server-Side)

Use a server-side call to the IDCS token endpoint. Here is the curl command for testing:

# First, encode the Client ID and Secret as Base64
echo -n "<Client-ID>:<Client-Secret>" | base64

# Then request a token using Resource Owner Password Credentials
curl --request POST \
  --url https://<IDCS-domain>.identity.oraclecloud.com/oauth2/v1/token \
  --header 'Authorization: Basic <base64-encoded-clientID:clientSecret>' \
  --header 'Content-Type: application/x-www-form-urlencoded;charset=UTF-8' \
  --data 'grant_type=password&username=<service-account-user>&password=<password>&scope=<scope-from-step-B2>'

A successful response returns a JSON object containing access_token, token_type, expires_in, and optionally refresh_token.

ℹ️
Token expiry: OAC access tokens default to 100 seconds. For embedding, you typically want to extend this. You can modify the token expiry on the IDCS application that corresponds to your OAC instance using the IDCS SCIM API (PATCH to /admin/v1/Apps/<AppID> with accessTokenExpiry and refreshTokenExpiry values). See Reference for the exact curl command.
Step B-5

Inject the Token into the Embedding Page

Once you have the access token, inject it into your HTML page using the setEmbeddingConfig API. This tells the embedding framework to use token-based auth instead of session cookies.

<oracle-analytics-assistant
  id="oaAssistant"
  src-data="[&quot;XSA('user'.'dataset')&quot;]">
</oracle-analytics-assistant>
<script>
  const token = '<access-token-from-server>';

  oracle.oa.embedding.ready().then((oApp) => {
    oApp.setEmbeddingConfig({
      oSecurityConfig: {
        eSecurityType: "token",
        oOptions: {
          credentials: "",
          tokenAuthFunction: new Function("return '" + token + "'")
        }
      }
    });
    oApp.applyBindings();
  });
</script>
❌
For testing only: The code above hardcodes the token in the HTML. In production, the token should be fetched dynamically from a secure server-side endpoint (see Step B-6). Never commit tokens to source control.
Step B-6

Production Pattern: Server-Side Token Endpoint

In production, you need a server-side component that generates and serves tokens. Oracle provides two recommended approaches:

Option 1: OCI Functions + API Gateway

  • Deploy an OCI Function (Node.js or Python) that calls the IDCS token API with the stored Client ID/Secret.
  • Expose the function via OCI API Gateway with an endpoint like https://api.yourcompany.com/oac-token.
  • The API Gateway can add authentication (API keys, OAuth) and rate limiting.
  • Your embedded page calls this endpoint to retrieve a fresh token before initializing.

Option 2: Your own backend API

  • Add an endpoint to your existing application server (Node.js, Python Flask/FastAPI, Java Spring, etc.) that makes the IDCS token call.
  • Return only the access_token to the client.

Here is a conceptual example of the client-side code with dynamic token fetching:

<script>
  async function initAssistant() {
    // Fetch token from your secure server endpoint
    const resp = await fetch('https://api.yourcompany.com/oac-token');
    const data = await resp.json();
    const token = data.access_token;

    const oApp = await oracle.oa.embedding.ready();
    oApp.setEmbeddingConfig({
      oSecurityConfig: {
        eSecurityType: "token",
        oOptions: {
          credentials: "",
          tokenAuthFunction: () => token
        }
      }
    });
    oApp.applyBindings();
  }

  initAssistant();
</script>
Step B-7

Handle Token Refresh

Access tokens have limited lifetimes. To prevent the embedded Assistant from failing mid-session, implement a token refresh strategy:

  1. When requesting the initial token, include offline_access in the scope to receive a refresh token alongside the access token.
  2. Store the refresh token securely on the server side (never expose it to the client).
  3. Before the access token expires, use the refresh token to obtain a new access token by calling the IDCS token endpoint with grant_type=refresh_token.
  4. Return the new access token to the client and re-inject it.

Alternatively, for simpler setups, you can set a longer token expiry (e.g., 3600 seconds) and have the page periodically fetch a fresh token from your server endpoint. The tokenAuthFunction callback is invoked each time the embedding framework needs a token, so you can make it return a dynamically refreshed value.

Decision Guide

Choosing the Right Model

Criteria Browser-Based (Session) Token-Based (OAuth 2.0)
User login required? Yes — user must have active OAC session No — token is obtained server-side
OAC account per user? Yes — every viewer needs an OAC account No — a single service account can serve all viewers
Third-party cookie issue? Yes — browsers may block cross-domain cookies None — no cookies are used for auth
Per-user security (row-level)? Yes — each user's identity is preserved Depends — token identity determines access; for per-user security, use authorization code grant
Setup complexity Low — just Safe Domains Medium-High — IDCS app, token endpoint, refresh logic
Best for Internal portals with SSO Public sites, kiosks, external portals
Oracle recommendation Good for internal / dev Recommended for production
💡
Hybrid approach: In enterprise deployments, it's common to use browser-based auth for internal users (who already have SSO) and token-based auth for external-facing embedded analytics — both can coexist across different pages.
Required for Both Models

Safe Domains Configuration

Regardless of which authentication model you choose, you must register your embedding page's origin in OAC's Safe Domains. This is OAC's CORS allowlist — it tells OAC which external origins are permitted to embed analytics content.

What to Register

  • Local development: http://localhost:8000 (or whichever port your local server uses).
  • Staging: https://staging.yourcompany.com
  • Production: https://portal.yourcompany.com

Resource Types to Enable

When adding a domain, OAC lets you select which resource types to allow. For embedding, you must at minimum enable:

  • Embedding — allows <oracle-analytics-assistant> and <oracle-dv> tags to render.
  • Scripts — allows the page to load embedding.js.
⚠️
After adding a Safe Domain, users must sign out and sign back in to OAC for the change to take effect. During development, refreshing the page is usually sufficient since no cached session is involved.

Vanity URLs

If third-party cookie blocking is a concern, configuring a vanity URL for OAC can help. A vanity URL maps OAC to a subdomain under your organization's domain (e.g., analytics.yourcompany.com). When the embedding page and OAC share the same top-level domain, browser cookie restrictions are typically not triggered.
